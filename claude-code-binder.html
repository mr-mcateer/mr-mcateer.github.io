Module 2: Agent Harnesses

Last verified: April 2, 2026 | Source: Anthropic engineering blog, LangChain

Definition VERIFIED

An agent harness is everything that wraps around the LLM that is not the LLM itself: the system prompt, hooks, available tools, permission modes, memory management, context compaction parameters, and execution environment.

LLM = intelligence. Harness = what makes it useful. Claude is the model. Claude Code is the harness.

Harness Components

ComponentWhat It Controls
System PromptCLAUDE.md files, injected at top of context window
ToolsBash, Read, Write, Edit, Grep, Glob, Web tools, MCP servers
HooksLifecycle automation: PreToolUse, PostToolUse, SessionStart, etc.
Permission ModesPlan mode, default, accept edits, auto mode, bypass permissions
MemoryContext compaction, autocompact buffer, conversation history
ParametersMax turns, token limits, model selection

Module 3: Parallelization & Multi-Agent Patterns

Last verified: April 2, 2026 | Sources: Microsoft Azure, MindStudio, OpenReview, Anthropic docs

Why Parallelize

Time

Serial: 4 steps x 5 min = 20 min. Parallel: 5 min concurrent + 2 min synthesis = 7 min. 65% faster.

Quality

Model performance degrades as context grows. Sub-agents stay in the "zone of good" with short, clean context windows.

Coverage

Stochastic outputs: 5 runs produce ~2.5x unique answers vs. 1 run. Outlier ideas surface that a single agent would miss.

Cost

Research with Sonnet ($3/MTok), synthesize with Opus ($5/MTok). ~60% savings on research tokens.

Pattern 1: Fan-Out / Fan-In VERIFIED

[Research Query] | +------------+------------+ | | | [Sonnet Agent 1] [Sonnet 2] [Sonnet N] <-- Fan Out (cheap, parallel) | | | +------------+------------+ | [Opus Synthesizer] <-- Fan In (smart, single) | [Final Answer]

Use when: Research tasks, API evaluation, competitor analysis, any "find the best X" query.

Documented by: Microsoft Azure Architecture Center, Kinde, multiple practitioners.

Pattern 2: Stochastic Consensus VERIFIED

Same prompt (slightly varied framing) --> N agents independently Agent 1: [A, B, C] Consensus count: Agent 2: [B, C, E] A: 2 B: 3 C: 3 (HIGH CONFIDENCE) Agent 3: [A, B, Q] E: 1 Q: 1 (OUTLIERS -- investigate)

Use when: Decision-making, ranking options, filtering hallucinations, surfacing high-variance ideas.

Documented by: MindStudio ("Stochastic Multi-Agent Consensus"), Kinde ("LLM Fan-Out 101").

Pattern 3: Multi-Round Debate VERIFIED

Time 1: Each agent generates independently Time 2: Each agent sees ALL other responses, refines Time 3: Further refinement with cross-pollination ... Time N: Convergence on nuanced, battle-tested solutions

Use when: Nuanced analysis, architecture decisions, anything where "it depends" is the first answer.

Documented by: Edoardo Schepis (Medium series), academic papers on OpenReview.

Pattern 4: Pipeline (Sequential Specialist Handoff)

[Dev Agent] --> [QA Agent] --> [Test Agent] | | | Fresh context Fresh context Fresh context Incentive: Incentive: Incentive: Build fast Find bugs Break things

Use when: Any build-then-verify workflow. Different incentives prevent blind spots.

Recommended Orchestration Architectures

Parent + Researchers + QA (Recommended)

Parent (Opus): orchestrates, builds
Researchers (Sonnet x N): fan-out research
QA (Opus): fresh context, finds bugs
Best balance of quality, speed, cost.

Developer + QA (Lean)

Dev (Opus): plans and builds
QA (Opus): fresh spawn after each feature, zero context bias
Simpler, fewer compound probability issues.

Agent Teams Feature VERIFIED

Status: Experimental. Requires CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS env var.
How it works: One session = team lead. Coordinates multiple independent teammates. Each has own context window. Can communicate via team lead.
Sub-agent model overrides: Supported via model field in SKILL.md frontmatter.

Module 4: Auto Research (Karpathy Framework)

Last verified: April 2, 2026 | Source: github.com/karpathy/autoresearch, Shopify/liquid PR #2056

The Three Prerequisites

Auto research works when -- and only when -- you have all three:

#RequirementExample (Website)Example (Cold Email)
1Metric to optimizeGoogle Lighthouse scoreReply rate %
2Change method (fast)Modify HTML/CSS/JSModify email copy
3Assessment (fast)Run Lighthouse test (~30s)Wait for replies (~1hr)
Speed math: If change + assess = 60 seconds, you get 1,440 experiments/day. At 2% success rate with 1% improvement each = 1.01^30 = 34% improvement per day.

The Loop VERIFIED

+---> [Hypothesize] ---> [Execute Change] ---> [Assess] | | | +--- improvement? ---> YES: Keep, log success | | NO: Revert, log failure | | +----------+ Research log accumulates: what worked, what didn't, why. Each cycle starts with full history -- search space shrinks.

Module 8: Workspace Organization

Last verified: April 2, 2026 | Note: These are personal conventions, not official Anthropic recommendations

Core Structure

business/ .env # API keys (gitignored) CLAUDE.md # Local system prompt .claude/ skills/ # Reusable SOPs settings.json # Hooks, permissions active/ # All generated output (never pollute root) tmp/ # Ephemeral files (hidden) model-chat/ # Skill-specific output dirs research/ web-design/ clients/ client-a/ .env # Client's API keys CLAUDE.md # Client context (/init + manual) .claude/skills/ # Client-specific skills client-b/ ... personal/ # Separate workspace, different VS Code color .env CLAUDE.md .claude/skills/ active/ health/ # Domain-specific subfolders citizenship/ ...

Key Principles

  • Never pollute root. All generated files go in active/ subdirectories.
  • Skills specify their output paths. Each SKILL.md declares where its output lives.
  • Client folders mirror business structure. Each has its own .env, CLAUDE.md, skills.
  • Periodically clean up. Ask Claude to organize loose files in active/.
  • Color-code workspaces. VS Code .vscode/settings.json with different title bar colors per workspace type.
  • Cross-workspace skill access. Add a line to CLAUDE.md: "Client skills live in clients/[name]/.claude/skills/ -- read from there when working on that client."

Alternative Approaches

  • Project-based: company/projects/{website, cro, lead-gen}
  • GitHub-synced: nightly push, daily clone, cloud backup
  • CRM-as-JSON: store customer data in workspace, sync to calendar
  • Or just ask Claude to design one for your needs

Module 9: Security -- The 80/20 That Actually Matters

Last verified: April 2, 2026 | Sources: OWASP, academic papers, Supabase docs

Guiding Principle

"Everything is hackable. It's always just a question of how hackable." The goal is not perfection -- it's making your attack surface more expensive to breach than the value of what's behind it. A fence and a camera, not Fort Knox.

The Five Low-Hanging Threats

1. Plain-Text API Keys in Conversation Logs VERIFIED

Risk: All Claude Code conversations stored at ~/.claude/projects/ in JSONL format. Any API key pasted in chat = stored in plain text on disk.
Fix: Store ALL secrets in .env. Reference by variable name only. Never paste raw keys into chat. Add .env to .gitignore.

2. Hallucinated Package Names (Supply Chain) VERIFIED

Risk: AI models hallucinate package names ~19.7% of the time (academic study: "We Have a Package for You," 2024). Attackers register these misspelled names with malware.
Fix: Audit dependency list after every npm install. Ask Claude: "Are all packages here legitimate, verified, and actively maintained?" Check for unfamiliar names before running.

3. Database Row-Level Security (RLS) VERIFIED

Risk: Supabase does NOT enable RLS by default. Without it, any user with a public key can read, write, and delete every row in your database.
Fix: Enable RLS on every table. Immediately. No exceptions.

4. Public-Facing Agent Servers

Risk: Any URL on a VPS IP range is continuously scanned by bot farms testing for vulnerabilities. Open ports, unpatched services, and exposed agent interfaces are found within hours.
Fix: Never expose agent interfaces directly. Use authenticated proxies (Telegram connectors, etc.). Don't give public agents access to sensitive data (passports, SSNs, credentials).

5. Credit Card Data

Risk: Storing card numbers anywhere = PCI compliance nightmare + massive liability. AI agents will log them in conversation history.
Fix: Use Stripe or equivalent payment processor. Never touch card numbers. Period.

Security Audit Workflow

  1. Build your feature / app / system
  2. Open a new, clean conversation (no development context bias)
  3. Paste a security audit prompt covering: secret scanning, dependency audit, RLS check, input validation, OWASP Top 10 for LLMs
  4. Review findings, then open another new conversation to implement fixes (avoids audit bias)
  5. Optionally: run the audit with a different model (Codex, Gemini) for provider diversity
