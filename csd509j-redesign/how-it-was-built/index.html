How This Site Was Built

A complete school district website — 10 pages, 1,650 lines of CSS, 13 school profiles — built by coordinated AI agents in a single session.

One Person. Many Agents. Parallel Execution.

Imagine a construction site. A foreman does not lay every brick, pull every wire, and weld every joint alone. Instead, the foreman coordinates — dispatching electricians, plumbers, and framers to work simultaneously on independent tasks. The building goes up fast because the work happens in parallel.

Agentic coding applies the same principle to digital work. Instead of one person writing one file at a time, an AI orchestrator dispatches specialized agents to work simultaneously on independent tasks. One agent writes CSS. Another structures data. A third builds the contact page while a fourth builds the schools page. They share a common blueprint — the design system — but their work is independent. No agent waits for another.

This is not AI replacing a human. This is AI multiplying the capacity of a single person. One teacher. One evening. A complete website.

The bottleneck in digital work has always been sequential execution. One person, one file, one task at a time. Agentic coding breaks that constraint.

The Architecture of Parallel Agent Work

Four phases, from blank directory to deployed site

Phase 1

Foundation (Sequential)

Before anything can run in parallel, the shared infrastructure must exist. Four agents worked in sequence to lay the groundwork that every subsequent page would depend on.

A1 Design system — CSS tokens, 32 component classes
A2 JavaScript modules — nav, search, animations
A3 Data files — schools, departments, navigation
A4 Deployment pipeline — GitHub Actions config
Why sequential? These are dependencies. The CSS must exist before pages can use its classes. The data files must exist before pages can render school profiles. Everything else builds on this foundation.
Phase 2

Pages (Parallel)

This is where the approach pays off. With the foundation in place, eight agents fire simultaneously. Each agent builds one page. Each page uses the same CSS, the same JavaScript, the same footer. No agent depends on another agent's output.

B1 About page
B2 Schools page + filter logic
B3 Families page + FAQ accordion
B4 Community page
B5 Staff page
B6 Departments page + card grid
B7 News page
B8 Contact page + accordion directory
Why parallel? Each page is independent. They share a common blueprint — same CSS, same JS, same nav structure — but no page reads from or writes to another page. This independence is what makes parallelism safe and correct.
Phase 3

Quality Audit

A fresh agent — one that did not write any of the code — reads all 10 pages from scratch. It checks structural consistency, path correctness, semantic HTML validity, CSS class usage, and cross-page link integrity. Think of it as an independent code reviewer with no ego and no fatigue.

C1 Full audit: found and fixed 8 issues across 176KB of HTML
Phase 4

Images, SEO, Deploy

Final production hardening. All images self-hosted to eliminate external CDN dependencies. Schema.org structured data added for search engines. Site deployed via GitHub Pages through a GitHub Actions pipeline. Zero build tools. Zero runtime dependencies.

D1 Self-host 59 images, optimize paths
D2 Schema.org + meta tags
D3 GitHub Pages deployment via Actions

The Implications for Digital Work

“The question is no longer whether one person can do the work of a team. The question is what that person chooses to build.”

Capacity

Capacity Multiplication

One person can now do the work of a small team — not by working faster, but by orchestrating parallel operations. A single evening produces what traditionally takes a week. The human contribution shifts from execution to judgment: knowing what to build, what quality looks like, and when the output meets the standard.

Quality

Quality Through Specialization

Each agent focuses on one task. The CSS agent only thinks about design systems. The schools page agent only thinks about filter logic and school data. Specialization produces better output than context-switching. A human juggling nine files introduces inconsistencies. Nine agents, each with a single focus, do not.

Access

Accessible to Non-Engineers

The orchestrator — the human — does not need to write CSS or JavaScript by hand. They need to understand what good output looks like and how to direct agents toward it. This is project management and editorial judgment, not programming. A teacher can build what previously required a developer.

Beyond Websites

The same pattern applies to any digital information work

The plan-parallelize-audit pattern that built this website is not specific to web development. It is a general framework for any work that can be decomposed into independent tasks.

Curriculum development: write 16 lesson plans simultaneously, each agent specializing in a single topic, then audit them for consistency and pedagogical alignment. Data analysis: query multiple datasets in parallel, each agent focused on one question, then synthesize the findings. Documentation: generate and cross-reference docs for an entire system in one pass. Communication: draft personalized messages to dozens of stakeholders, each tailored to its audience.

The constraint was never computing power. It was the serial nature of human attention. One person can only hold one file, one conversation, one analysis in their head at a time. Agentic orchestration removes that bottleneck. The human provides direction and quality standards. The agents provide throughput.

Every school district has one person who could do twice as much — if the tools let them work in parallel. This is what that looks like.

Under the Hood

Technical details for those who want to look deeper

Design System

Built on CSS custom properties with an 8pt spacing grid. Typography uses three font stacks: Playfair Display for display headings, Inter for body text, and JetBrains Mono for data, labels, and code. Shadows follow an Apple-inspired elevation system (xs through xl). The frosted-glass navigation uses backdrop-filter: blur(20px) saturate(180%) for the characteristic translucency effect. Responsive breakpoints at 1024px, 768px, and 480px. 32 reusable component classes — cards, stats bars, callouts, accordions, filter tabs, buttons, grids — ensure visual consistency without duplicating styles across pages.

Data Architecture

School data is stored in a JSON structure with consolidation-resilient status fields, allowing schools to be marked as closing, merging, or transitioning without removing their records from the site. The department directory is similarly data-driven: a single JSON file powers the departments page card grid, the contact page accordion, and the search index. The navigation structure is defined once and rendered identically across all 10 pages. This single-source-of-truth approach means updating a school name or phone number propagates everywhere automatically.

Accessibility

The site targets WCAG 2.1 AA compliance. Implementations include: skip-to-main-content link on every page, ARIA labels on all interactive regions and landmarks, full keyboard navigation for accordions, search overlay (Cmd/K to open, Escape to close), and mobile navigation drawer. A prefers-reduced-motion media query disables all animations and transitions for users who request it. All text-to-background combinations maintain a minimum 4.5:1 contrast ratio. A print stylesheet strips navigation and chrome, sets readable fonts, and appends URLs to printed links.

Search

Client-side search implemented as a Cmd/K (or Ctrl+K on Windows) overlay. The search index contains 35 entries spanning all pages, departments, schools, and key resources. Input is debounced at 120ms to prevent excessive re-rendering during fast typing. Matching is case-insensitive substring search across title and description fields, with matched terms highlighted using <mark> elements. Results are categorized by type (page, school, department) with monospaced category labels. The entire search system is a single vanilla JS file with zero dependencies.

Deployment

Deployed to GitHub Pages via a GitHub Actions workflow triggered on push to the main branch. The site is entirely static — plain HTML, CSS, and vanilla JavaScript. No build step, no bundler, no package.json, no node_modules. This eliminates an entire category of maintenance burden: there are no dependencies to update, no security patches to apply, no build tools to break. The site loads in under 1 second on a median mobile connection because there is nothing to compile, transpile, or hydrate. It is just files served directly from a CDN.
